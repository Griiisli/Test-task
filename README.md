# Test-task

Юзер-сценарії (Playwright) для воронки квізу `dvrcres0lve.dev` → редирект на YourForms.

## Структура

```
playwright.config.js          # конфіг (testDir, timeout, workers)
Tests/
  scenario-1_happy-path.spec.js   # повний прохід квізу до редіректу
  scenario-2_redirect-url.spec.js # валідація URL редіректу
  scenario-3_cta-buttons.spec.js  # всі CTA-кнопки ведуть на /quiz
  helpers/quiz.helper.js          # логіка проходження квізу
```

## Чому саме ці 3 сценарії (покриття бізнес-логіки)

Бізнес — це **конверсійна воронка лідогенерації**: відвідувач із лендингу проходить
квіз-кваліфікацію і передається партнеру (YourForms), який монетизує лід. Виручка
залежить рівно від одного: чи доходить користувач від входу до коректної передачі.

У цієї воронки є **три незалежні точки відмови, кожна з яких самостійно обнуляє
конверсію**, — і кожен сценарій закриває рівно одну з них:

1. **Scenario 1 — Happy Path (чи воронка взагалі працює).**
   Наскрізний шлях лендинг → квіз → редирект. Якщо користувач фізично не може
   дійти до передачі — конверсія = 0. Покриває рендер і перехід усіх типів кроків
   (options, dropdown штат/графство, форма імені, проміжні екрани, `analyzing`)
   та збір даних. Це smoke-тест: впав він — решта не має сенсу.

2. **Scenario 2 — Redirect contract (чи передача *коректна*).**
   Навіть якщо квіз пройдено, неправильний домен/шлях або втрачений параметр
   сесії `h` означає, що партнер не отримує лід → конверсія = 0 при формально
   «робочій» воронці. Перевіряє саме **контракт передачі** вниз по ланцюгу.

3. **Scenario 3 — CTA entry points (чи у воронку взагалі заходять).**
   Трафік іде з кількох CTA по сторінці; зламана кнопка тихо вбиває джерело входу,
   і happy-path цього не зловить (він ходить лише через Hero). Дешевий регрес-щит
   для верхівки воронки.

**Чому це оптимально:**
- **Мапиться на реальні бізнес-ризики, а не на сторінки.** Три сценарії = три
  різні способи втратити гроші (не зайшов / не дійшов / зайшов не туди), а не
  механічний обхід UI.
- **Шарування за вартістю/глибиною.** Scenario 3 — дешевий і широкий (вхід),
  Scenario 1 — глибокий золотий шлях (потік), Scenario 2 — точність контракту
  (передача). Разом — тонкий, але повний вертикальний зріз воронки без дублювання.
- **Максимум покриття ризику на секунду прогону.** Важку воронку проходимо
  мінімально (один раз на сценарій через `beforeAll`), входи — легкі й
  параметризовані.
- **Black-box, від імені користувача.** Асерти на спостережувані результати
  (потрапив на квіз / пройшов / зредіректило з сесією), а не на деталі реалізації —
  стійко до рефакторингу фронтенду.

> **Межі покриття.** Ці 3 сценарії оптимально закривають **revenue-critical ядро**
> (happy path + контракт передачі). Глибші edge-cases — валідація-гейтинг,
> resume після refresh, мобільний viewport, альтернативні гілки квізу — це
> окремий, комплементарний шар, який доцільно додати поверх цього ядра.

## Вимоги

- Node.js 18+
- Playwright 1.58+ (ставиться через `npx`, окремий `package.json` не потрібен)

## Встановлення браузерів

Перед першим запуском треба завантажити браузер Chromium:

```bash
npx playwright install chromium
```

## Запуск тестів

Усі тести:

```bash
npx playwright test
```

Окремий сценарій:

```bash
npx playwright test scenario-1          # тільки happy path
npx playwright test scenario-2 scenario-3
```

Один конкретний тест-кейс:

```bash
npx playwright test -g "TC-1.4"
```

## Корисні прапорці

```bash
npx playwright test --headed     # показувати браузер (за замовчуванням headless:false)
npx playwright test --debug      # покроковий дебаг (Playwright Inspector)
npx playwright show-report       # відкрити HTML-звіт останнього запуску
```

## Примітки

- Тести працюють проти **живого** сайту `https://dvrcres0lve.dev/`, тож потрібен інтернет.
- `workers: 1` — тести виконуються послідовно: кілька паралельних браузерів проти живого
  сайту конкурують за ресурси й повільні API-запити, через що кроки стають нестабільними.
- `timeout: 120000` — повний прохід воронки триває ~50 секунд.
- `retries: 1` — живий сайт інколи роняє браузер (`Target page closed`); один retry
  прибирає цей періодичний збій. Trace зберігається `on-first-retry`.
- У dropdown штату/графства захардкоджено **Alabama** → **Autauga** для детермінованого вибору.

## Знайдені проблеми

Під час тестування воронки виявлено такі проблеми:

1. **Нестабільний searchable-dropdown (штат → графство, крок `q5`).**
   Після вибору штату підвантажується dropdown графства, але вибір опції не
   завжди реєструється — кнопка Continue лишається `disabled`, і користувач
   застрягає на кроці. У тестах обійдено хардкодом значень (Alabama → Autauga)
   з вводом тексту для фільтрації списку.

2. **Непослідовна поведінка кроку `analyzing`.**
   Інколи крок сам редіректить на YourForms, інколи лишає активну кнопку
   Continue, яку треба натиснути вручну. Час до редіректу теж нестабільний
   (від кількох секунд до десятків). Це ускладнює автоматизацію та може
   збивати з пантелику живих користувачів.
